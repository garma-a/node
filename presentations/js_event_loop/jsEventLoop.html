JavaScript Event Loop

Deep Dive into Async Runtime Engine

Understanding Call Stack, Web APIs, Microtasks, and Macrotasks

JavaScript Runtime Architecture

JavaScript is Single-Threaded

Only one function executes at a time

But... JavaScript feels asynchronous 🤔

The browser's Web APIs + Event Loop create the illusion

Solution: Event Loop manages async operations

JavaScript Runtime Architecture

CALL STACK

Tracks function execution. LIFO structure (Last-In-First-Out).

WEB APIS

setTimeout, fetch, DOM events - managed by browser.

TASK QUEUE

Macrotasks: setTimeout, setInterval, I/O events.

MICROTASK QUEUE

Promises, async/await, queueMicrotask(), MutationObserver.

Event Loop: Execution Flow

1. Execute all synchronous code (Call Stack)

↓

2. Process ALL microtasks (Promises, async/await)

↓

3. Process ONE macrotask (setTimeout, I/O, etc.)

↓

Go back to step 2 (repeat until queues empty)

Call Stack: The Execution Context

console.log('1');

function foo() { console.log('2'); }

foo();

console.log('3');

Stack State (LIFO):

console.log('1')
foo()
console.log('2') [inside foo]

↓ (pops after each)

console.log('3')

Output: 1 → 2 → 3

Example: setTimeout Callback

console.log('Start');

setTimeout(() => console.log('Async'), 0);

console.log('End');

Execution Steps:

  • 1. 'Start' logged (sync, call stack)
  • 2. setTimeout offloaded to Web API timer
  • 3. 'End' logged (sync, call stack)
  • 4. Call stack empty → Event Loop checks queues
  • 5. Timer fires → callback goes to Task Queue
  • 6. 'Async' logged (task queue executed)

Output: Start → End → Async

Microtasks vs Macrotasks

🟢 MICROTASKS (HIGH PRIORITY)

  • Promise.then/catch/finally
  • async/await
  • queueMicrotask()
  • MutationObserver
  • process.nextTick (Node.js)

✓ Executed BEFORE next macrotask

✓ All processed in one loop

🟡 MACROTASKS (LOWER PRIORITY)

  • setTimeout/setInterval
  • setImmediate (Node.js)
  • I/O operations
  • DOM events
  • requestAnimationFrame

✓ One processed per loop

✓ Lower priority than microtasks

Example: Microtask Priority

console.log('Start');

setTimeout(() => console.log('Timeout'), 0);

Promise.resolve().then(() => console.log('Promise'));

console.log('End');

Output: Start → End → Promise → Timeout

Why? Promise callbacks (microtasks) execute BEFORE setTimeout callbacks (macrotasks), even with 0ms delay!

Promises: The Microtask Gateway

Promise States:

✓ Pending → Fulfilled / Rejected

Promise Resolution Flow:

1. Promise resolves → .then() callback queued

2. Callback added to Microtask Queue

3. Executes BEFORE any setTimeout callbacks

Microtasks ensure Promise resolution happens immediately after current execution

Async/Await: Syntactic Sugar for Promises

async function:

• Always returns a Promise

• Return value wrapped in Promise

• Errors become rejected promises

await operator:

• Pauses execution until Promise settles

• Callback added to Microtask Queue

• Actually: returns Promise → microtask for continuation

⚠️ Key: Both async/await and Promises use Microtask Queue!

Async/Await Execution Model

async function fetchData() {

let result = await fetch('api/data');

return result;

}

Under the hood:

  • 1. fetch() returns Promise (macrotask: I/O operation)
  • 2. await pauses async function
  • 3. Creates microtask for continuation after fetch settles
  • 4. Continuation (return result) runs as microtask

Real Example: Complex Execution Order

console.log('1');

setTimeout(() => console.log('2'), 0);

Promise.resolve()

.then(() => { console.log('3'); setTimeout(() => console.log('4'), 0); })

.then(() => console.log('5'));

console.log('6');

Output: 1 → 6 → 3 → 5 → 2 → 4

Step-by-Step Breakdown

Phase 1 (Sync Code): 1, 6 logged

Phase 2 (All Microtasks): Promise callbacks execute

• First .then() → logs 3, queues setTimeout('4')

• Second .then() → logs 5

✓ Both microtasks done in one loop!

Phase 3 (First Macrotask): logs 2

Phase 4 (Microtasks again): None

Phase 5 (Second Macrotask): logs 4

Node.js Event Loop Phases

1. timers → 2. pending callbacks → 3. idle/prepare

4. poll (I/O) → 5. check → 6. close callbacks

⭐ Microtasks execute between EVERY phase!

Node.js specific: process.nextTick() runs BEFORE microtasks

Microtask Starvation Problem ⚠️

If microtask queue continuously adds tasks:

Macrotasks NEVER execute! 🔴

Example: Recursive Promise.then() prevents setTimeout from running

✗ Don't create infinite Promise chains in production code

✓ Ensure microtasks eventually complete

Performance Implications

✓ Long Microtasks:

All other microtasks delayed, but UI rendering delayed further (good for batch updates)

✗ Heavy Macrotasks:

UI blocking! Microtasks also blocked. Use requestAnimationFrame or break into chunks

Tip: Use requestAnimationFrame for smooth animations instead of nested setTimeout

Debugging Async Code

console.trace() - Shows call stack at specific point

debugger; - Breakpoint in DevTools

async Stack Traces - DevTools shows Promise chain

Performance tab - Visualize event loop phases

Chrome DevTools: Sources tab → Pause on exceptions → Async stack

Best Practices

✓ Use async/await over .then() chains

More readable, easier to debug

✓ Batch DOM updates in single Promise

Use microtask queue for efficient rendering

✓ Understand execution order

Critical for race condition prevention

✗ Avoid deeply nested Promises

Use async/await or Promise.all() for clarity

Common Async Pitfalls

❌ Unhandled Promise Rejection

Silent failures, hard to debug

❌ Race Conditions

Multiple async operations modify shared state

❌ Memory Leaks from Callbacks

Event listeners not cleaned up after async operations

Key Takeaways

1. JS is single-threaded, Event Loop enables async behavior

2. Call Stack → Microtask Queue → Macrotask Queue (repeat)

3. Microtasks (Promises, async) run BEFORE macrotasks (setTimeout)

4. async/await = syntactic sugar for Promises (both use microtasks)

5. Understand order to prevent race conditions & optimize performance
